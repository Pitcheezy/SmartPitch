## 작업 1 — EarlyStopping 도입

**커밋**: `3a5d4c4` (03-29)
**파일**: `src/model.py`

### 목적
epochs를 5→20으로 늘릴 예정이므로, 과적합을 방지할 안전장치가 필요했다.
기존에는 val_loss가 올라가도 학습을 끝까지 돌려서 과적합된 모델이 저장될 위험이 있었다.

### 수정 전
```python
def train_model(self, epochs: int = 50, hidden_dims=..., dropout_rate=...):
    best_val_loss = float('inf')
    for epoch in range(1, epochs + 1):
        # ... 학습/검증 ...
        if val_loss < best_val_loss:
            best_val_loss = val_loss
            torch.save(self.model.state_dict(), "best_transition_model.pth")
```

### 수정 후
```python
def train_model(self, epochs: int = 50, ..., patience: int = 5):
    best_val_loss = float('inf')
    patience_counter = 0
    for epoch in range(1, epochs + 1):
        # ... 학습/검증 ...
        if val_loss < best_val_loss:
            best_val_loss = val_loss
            patience_counter = 0
            torch.save(self.model.state_dict(), "best_transition_model.pth")
        else:
            patience_counter += 1
            if patience_counter >= patience:
                print(f"Early stopping: epoch {epoch}에서 중단")
                if wandb.run:
                    wandb.log({"early_stop_epoch": epoch})
                break
```

### 기술적 이유
- patience=5: val_loss가 5 epoch 연속 개선되지 않으면 중단
- "최고 모델 저장"은 기존 로직 그대로 유지 → EarlyStopping은 추가 학습을 막을 뿐, 저장 시점은 바꾸지 않음
- `early_stop_epoch`을 W&B에 로깅 → 나중에 어느 epoch에서 멈췄는지 추적 가능

### 배울 점
- EarlyStopping과 "best model 저장"은 별개 관심사. 둘 다 val_loss 기준이지만 역할이 다르다: 하나는 "언제 멈출지", 하나는 "무엇을 남길지".
- patience 값을 하이퍼파라미터로 노출해야 나중에 W&B Sweep에서 탐색 가능.

## 작업 2 — pandas 3.0 호환 버그 수정

**커밋**: `d4ca474` (03-30)
**파일**: `src/model.py`

### 목적
`uv sync`로 설치한 pandas 3.0에서 `pd.get_dummies()`의 반환 dtype이 바뀌어 PyTorch 텐서 변환이 실패하는 버그 수정.

### 증상
```
RuntimeError: Can't call numpy() on Tensor with dtype object
# 또는
TypeError: can't convert np.ndarray of type numpy.object_
```

### 원인
pandas 2.x에서 `get_dummies()`는 `int64` (0/1) 컬럼을 반환했지만,
pandas 3.0부터 `bool` dtype을 반환한다.

```python
X_num   = pd.DataFrame({'balls': [3], 'strikes': [2]})   # int64
X_cat   = pd.get_dummies(df[['zone']])                    # bool (pandas 3.0)
X_encoded = pd.concat([X_num, X_cat], axis=1)             # 혼합 → object dtype!
```

int64 + bool을 concat하면 pandas가 공통 dtype을 찾지 못해 `object`가 되고,
이를 numpy 배열로 변환하면 `numpy.object_`가 되어 `torch.FloatTensor()`가 실패한다.

### 수정 전
```python
X_encoded = pd.concat([X_num.reset_index(drop=True), X_cat_encoded.reset_index(drop=True)], axis=1)
```

### 수정 후
```python
X_encoded = pd.concat([X_num.reset_index(drop=True), X_cat_encoded.reset_index(drop=True)], axis=1).astype(float)
```

### 기술적 이유
`.astype(float)`를 concat 결과에 한 번 호출하면 bool→float64, int64→float64로 통일된다.
get_dummies에 `dtype=float`을 인자로 줄 수도 있지만, concat 후 한 번에 처리하는 게 더 방어적.

### 배울 점
- **메이저 버전 업그레이드는 dtype 변경을 동반할 수 있다.** pandas 3.0의 `get_dummies()` bool 변경은 릴리스 노트에 명시돼있지만, 기존 코드에서 dtype을 명시하지 않으면 조용히 깨진다.
- PyTorch와 pandas를 연결하는 지점(numpy 변환)에서 dtype을 명시적으로 보장하는 습관이 필요.

## 작업 6 — hit_by_pitch 클래스 제거

**커밋**: `b49309c` (03-30)
**파일**: `src/model.py`

### 목적
진단 결과(작업 4)에서 hit_by_pitch가 전체의 0.3%이고 F1=0.000인 것을 확인.
이 클래스를 제거해서 5클래스→4클래스 체계로 전환.

### 추가된 코드 (위치가 중요)
```python
# _prepare_data() 내부, X_num/X_cat 구성 **전**에 배치
_hbp_mask = self.df['description'] != 'hit_by_pitch'
_removed  = (~_hbp_mask).sum()
if _removed > 0:
    self.df = self.df[_hbp_mask].reset_index(drop=True)
    print(f"  hit_by_pitch 제거: {_removed:,}건 → 남은 데이터: {len(self.df):,}건")
```

### 왜 X_num/X_cat 구성 전에 배치해야 하는가?

이것이 이 작업에서 가장 중요한 포인트다.

```
[잘못된 순서]
1. X_num 구성 (754,940행)
2. X_cat 구성 (754,940행)
3. hit_by_pitch 필터 (→ 752,740행)   ← X와 y의 행 수 불일치!

[올바른 순서]
1. hit_by_pitch 필터 (→ 752,740행)
2. X_num 구성 (752,740행)             ← 같은 행 수
3. X_cat 구성 (752,740행)             ← 같은 행 수
```

처음에 3번 위치에 넣었다가 `IndexError`가 발생해서 1번 위치로 이동했다.

### 결과
```
5클래스 → 4클래스 전환 후:
  macro avg F1: 0.381 → 0.477  (+0.096)
  ball F1:      0.669 → 0.668  (변화 없음)
  foul F1:      0.218 → 0.264  (+0.046)
  hit_into_play F1: 0.381 → 0.413  (+0.032)
```

### 배울 점
- **데이터 필터링의 위치는 로직의 정확성을 결정한다.** 필터를 어디에 두느냐에 따라 X와 y의 행 수가 맞거나 안 맞는다. "코드가 돌아가는 것"과 "정확한 것"은 다르다.
- 극소수 클래스 제거는 남은 클래스의 학습에 집중할 수 있게 해준다. 총 정확도는 비슷하지만 macro F1이 유의미하게 개선됐다.
- hit_by_pitch는 투수의 의도적 선택이 아니라 사고에 가깝다. 도메인 지식이 클래스 제거 결정을 뒷받침한다.

## 작업 7 — class-weighted CrossEntropyLoss

**커밋**: `bbd40c9` (03-30)
**파일**: `src/model.py`

### 목적
4클래스 전환 후에도 foul(22.9%)과 hit_into_play(12.4%)의 recall이 낮다.
소수 클래스에 더 높은 가중치를 줘서 균형 잡힌 학습 유도.

### 추가된 코드
```python
def train_model(self, ..., use_class_weights: bool = False):
    if use_class_weights:
        y_train_np = train_loader.dataset.y.numpy()
        counts = np.bincount(y_train_np, minlength=output_dim).astype(float)
        weights = 1.0 / np.where(counts > 0, counts, 1.0)
        weights = weights / weights.mean()        # 평균=1 정규화
        weight_tensor = torch.FloatTensor(weights).to(self.device)
        criterion = nn.CrossEntropyLoss(weight=weight_tensor)
    else:
        criterion = nn.CrossEntropyLoss()
```

가중치 계산 방식:
```
class_weight = (1 / class_count) / mean(1 / all_class_counts)
```

예시 (4클래스):
```
ball(267K)   → weight 0.41   (많으니까 줄임)
strike(218K) → weight 0.50
foul(172K)   → weight 0.63
hit_into_play(93K) → weight 1.17  (적으니까 키움)
```

### 결과
```
Exp3 ClassWeights vs Exp1 Baseline:
  val_acc: 57.3% vs 58.1%  (-0.8%p)
  foul F1: 0.204 → 0.265   (+0.061)
  hit_into_play F1: 0.381 → 0.413 (+0.032)
```

### 기술적 이유
- class-weighted loss는 소수 클래스의 gradient를 증폭시킨다. 다수 클래스(ball) 정확도를 약간 희생하고 소수 클래스(foul, hit_into_play) recall을 올린다.
- "정규화하여 평균=1로" 맞추는 이유: 가중치를 정규화하지 않으면 loss 스케일 자체가 바뀌어 learning rate와 EarlyStopping 기준이 영향받는다.

### 배울 점
- class-weighted loss는 **accuracy와 macro-F1 사이의 트레이드오프**다. 어떤 메트릭이 프로젝트에 더 중요한지에 따라 선택이 달라진다.
- 이 경험이 작업 8(실험 비교 기준 변경)의 직접적 원인이 된다.

## 작업 8 — load_from_checkpoint 아키텍처 불일치 수정 + 실험 비교 기준 변경

### Issue 1: load_from_checkpoint 아키텍처 불일치

**문제 상황**
```
universal_model_trainer.py에서 Exp1 [256,128,64]가 best로 선정됨

```

```
RuntimeError: Error(s) in loading state_dict:
  size mismatch for layers.0.weight: copying a param with shape [256, 40] from checkpoint,
  the shape in current model is [128, 40]
```

### Issue 2: 가중 vs 비가중 val_loss 비교 불가

**문제 상황**
```
Exp1 (unweighted): val_loss = 1.008
Exp3 (weighted):   val_loss = 1.097   ← 가중치가 loss를 증폭시키므로 스케일이 다름

min(val_loss) 기준 → 항상 unweighted가 유리 → weighted 실험이 절대 선정 안 됨
```

**수정**
```python
# 수정 전
best_result = min(results, key=lambda r: r["best_val_loss"])

# 수정 후
best_result = max(results, key=lambda r: r["final_val_acc"])
```

val_acc는 가중치와 무관하게 동일한 기준(맞춘 개수 / 전체)이므로 공정 비교 가능.

### 배울 점
- **모델 저장/로드는 가중치뿐 아니라 아키텍처도 포함해야 한다.** PyTorch의 `state_dict`는 파라미터 값만 담고, 모델 구조는 담지 않는다. 여러 아키텍처를 실험하면 이 문제가 반드시 발생한다.
- 해결 방법: (1) model_config JSON (이 프로젝트), (2) `torch.save(model)` 전체 저장 (pickle 의존성 문제), (3) ONNX export. JSON이 가장 투명하고 디버깅이 쉬움.
- **서로 다른 loss 함수의 값은 비교할 수 없다.** 이건 직관적이지만, 실험을 자동화하면 코드가 기계적으로 `min(loss)`를 선택하면서 놓치기 쉽다. 비교 메트릭은 반드시 loss-independent(accuracy, F1 등)이어야 한다.
- 이 두 이슈는 **코드 리뷰에서 발견**되었다. 학습+저장+로드+비교 경로를 처음부터 끝까지 테스트하지 않으면 개별 모듈은 잘 동작하는데 통합 시 깨지는 전형적인 통합 오류다.

## 핵심 교훈 (이틀간 반복된 패턴)

### 1. 진단 먼저, 개선은 그 다음
val_acc 57.9% 정체 → 바로 모델을 키우지 않고, confusion matrix로 **어디서 실패하는지** 먼저 확인.
진단 결과가 hit_by_pitch 제거, class-weighted loss 순서를 결정했다.

### 2. 위치(순서)가 정확성을 결정한다
hit_by_pitch 필터의 위치가 X_num 구성 전/후냐에 따라 IndexError 발생 여부가 갈렸다.
"어떤 코드를 쓰느냐"뿐 아니라 "어디에 두느냐"가 동등하게 중요하다.

### 3. 저장과 로드는 대칭이어야 한다
학습 시 `[256,128,64]`로 저장 → 로드 시 `[128,64]` 기본값 → size mismatch.
저장할 때 무엇을 기록하는지와 로드할 때 무엇을 읽는지가 정확히 대응해야 한다.

### 4. 서로 다른 loss 함수의 값은 비교할 수 없다
weighted loss는 스케일이 다르므로 `min(loss)`로 비교하면 항상 unweighted가 이긴다.
비교 메트릭은 loss-independent(accuracy, F1)이어야 한다.

### 5. 코드 리뷰는 통합 오류를 잡는다
개별 모듈 테스트에서는 발견 안 되는 이슈(아키텍처 불일치, loss 비교 불가)가
Cursor 리뷰에서 발견됐다. 전체 경로를 따라가는 시선이 필요하다.
